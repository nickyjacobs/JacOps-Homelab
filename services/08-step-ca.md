# step-ca

🇬🇧 English | 🇳🇱 [Nederlands](08-step-ca.nl.md)

step-ca is the homelab's internal ACME server. All services that need TLS request their certificates here via the standard ACME protocol, with short-lived certs that renew automatically. It runs as a native binary in an LXC container, accessible internally via `step-ca.jacops.local`.

## Why an internal ACME server

The manual OpenSSL CA previously in use (RSA 4096, two-year validity per cert) had three problems. A compromised service key gave an attacker two years of valid impersonation. Manual cert renewal does not scale at twenty services. And there was no mechanism to revoke a compromised cert.

step-ca addresses all three. Leaf certs are valid for 72 hours by default and renew automatically. A stolen cert becomes worthless within three days. Revocation is passive: short-lived certs simply expire. And the ACME provisioner lets Traefik and other clients request certs without manual intervention.

Let's Encrypt was not an option for internal traffic. Certificate Transparency logs expose internal domain names, and it introduces an external dependency for traffic that never leaves the cluster.

See [decisions.md](../docs/decisions.md) under "step-ca as internal ACME server" for the full reasoning.

## Architecture

```
ACME Client ─── HTTPS ──► step-ca (ACME server) ──► BadgerDB
(Traefik)                 :8443                     (cert state)
                          Intermediate CA key
                          (EC P-256, JWE encrypted)

                LXC Container (CT 164)
                ┌──────────────────────────────────────┐
                │  step-ca v0.30.2                     │
                │  ├─ ACME provisioner                 │
                │  ├─ Intermediate cert + key           │
                │  └─ Root cert (no root key!)         │
                └──────────────────────────────────────┘
                VLAN 40 (Apps)

Offline: USB drive CA-ROOT
         ├─ Root CA cert + key (encrypted)
         └─ Intermediate key backup (encrypted)
```

step-ca listens on port 8443 and answers ACME requests from clients on the network. The intermediate CA key lives on the LXC itself, encrypted with JWE. The root CA key is not on the LXC. It lives offline on a USB drive and is only connected when creating or renewing the intermediate.

No public tunnel, no Cloudflare. The service is only reachable via the local network or WireGuard.

## PKI

### Two-tier setup

step-ca runs a two-tier PKI: an offline root CA and an online intermediate CA. The root only signs the intermediate. The intermediate signs all leaf certs that clients request.

| Layer | Key type | Validity | Location |
|-------|----------|----------|----------|
| Root CA | EC P-256 | 10 years | Offline on USB drive CA-ROOT (FAT32) |
| Intermediate CA | EC P-256 | 5 years | CT 164, JWE-encrypted on disk |
| Leaf certs | EC P-256 | 72 hours default, 168 hours max | Generated by ACME clients |

EC P-256 is the step-ca default and offers the broadest compatibility with ACME clients, browsers and TLS libraries.

### Root CA

The root key lives on a USB drive labelled CA-ROOT (FAT32). The drive is only connected to the LXC when the intermediate needs to be renewed or re-signed. The rest of the time it sits in locked physical storage.

A backup of the root key is stored in Vaultwarden as `homelab/step-ca-root-key-passphrase` (the passphrase) and the key itself on a second USB drive at a different physical location.

### Intermediate CA

The intermediate key lives on CT 164 in the step-ca configuration directory. The file is encrypted with JWE (PBES2-HS256+A128KW / A256GCM). At startup, step-ca reads the password from `/etc/step-ca/password.txt` (chmod 600, owned by `step`).

A backup of the intermediate key lives on the same USB drive as the root key, plus in the weekly PBS backup of the container.

### Key encryption

All private keys (root and intermediate) are encrypted with JWE. The encryption scheme is PBES2-HS256+A128KW for key wrapping and A256GCM for content encryption. This is the default of step-ca's `step crypto` tooling.

The passphrases are stored in Vaultwarden:

| Secret | Vaultwarden path |
|--------|------------------|
| Root key passphrase | `homelab/step-ca-root-key-passphrase` |
| Intermediate key passphrase | `homelab/step-ca-intermediate-key-passphrase` |

## ACME provisioner

step-ca is configured with an ACME provisioner that supports the `tls-alpn-01` challenge type. Clients prove domain ownership by presenting a self-signed cert with a specific ALPN extension on port 443 of the requested domain.

| Setting | Value |
|---------|-------|
| Provisioner type | ACME |
| Challenge | `tls-alpn-01` |
| Default cert lifetime | 72 hours |
| Max cert lifetime | 168 hours |

72 hours as the default gives sufficient margin for a failed renewal (most ACME clients renew at two thirds of the lifetime, so after 48 hours). 168 hours as the maximum prevents clients from accidentally requesting long-lived certs.

## Why a software key instead of YubiKey

The original plan was to place the intermediate key on YubiKey PIV slot 9c (non-exportable, signing requires physical YubiKey plus PIN). In practice this is incompatible with automated ACME cert issuance: every signing operation would require physical interaction. With twenty services each renewing every 48 hours, that is not workable.

The software key with JWE encryption is the compromise. The key sits encrypted on disk and is unlocked at startup. The trade-off is that an attacker with root access to CT 164 could read the unlocked key from memory. The mitigation is systemd hardening, restricted network access and monitoring via the weekly PBS backup which provides an unmodified baseline.

## Security

### systemd hardening

step-ca runs as a dedicated `step` user with a strict systemd unit:

| Directive | Value |
|-----------|-------|
| `Type` | `notify` (sd_notify support since v0.29.0) |
| `User` | `step` |
| `NoNewPrivileges` | `true` |
| `ProtectSystem` | `full` |
| `ProtectHome` | `true` |
| `PrivateTmp` | `true` |
| `PrivateDevices` | `true` |
| `MemoryDenyWriteExecute` | `true` |

### File permissions

| Path | Permissions | Owner |
|------|------------|-------|
| `/etc/step-ca/` | 700 | `step:step` |
| `/etc/step-ca/password.txt` | 600 | `step:step` |
| Intermediate key | 600 | `step:step` |
| Root cert (public) | 644 | `root:root` |

The root cert is the only publicly readable file. Clients need it to verify the certificate chain.

### Passive revocation

step-ca does not run a CRL or OCSP responder. With a maximum lifetime of 168 hours for leaf certs, active revocation is unnecessary. A compromised cert loses its validity within three to seven days. This is the same model that cloud PKI systems like Google Certificate Authority Service use for short-lived workload certificates.

## Access

| Path | Purpose |
|------|---------|
| `https://step-ca.jacops.local:8443/acme/acme/directory` | ACME directory endpoint |
| `https://step-ca.jacops.local:8443/health` | Health check |

Both are only reachable via the local network or WireGuard. There is no public URL and no web UI. Administration happens via the `step` CLI on the LXC itself.

## Backup

The container is included in the weekly PBS backup job (Sunday 03:00, four weeks retention). This captures the full container filesystem including the intermediate key, the step-ca configuration and the BadgerDB state.

The root key lives offline on USB and falls outside the PBS backup. The recovery path for the root key is the second USB drive at a different physical location plus the passphrase in Vaultwarden.

## Migration from the manual CA

The existing `JacOps Homelab CA` (RSA 4096, manual OpenSSL workflow) remains trusted in the macOS system keychain until all services have migrated to step-ca. The migration order per service is: configure Traefik with the step-ca ACME endpoint, wait for cert renewal, remove old cert. After the last migration the old CA is removed from the trust store.
